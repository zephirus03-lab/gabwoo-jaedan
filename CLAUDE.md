# 갑우문화사 AX 전환

이 폴더는 갑우문화사의 AX(AI Transformation) 전환 작업 전체를 담는 최상위 프로젝트이다.
하위 폴더는 각 사업부 또는 개별 서브 프로젝트별로 별도의 CLAUDE.md를 가진다.

## 프로젝트 구조

```
gabwoo/                              # 갑우문화사 AX 전환 (umbrella)
├── CLAUDE.md                        # 본 파일 (전체 규칙 · 회사 정보)
├── 출판_생산 진행 현황/             # 출판사업부
│   └── CLAUDE.md
├── 패키지_생산진행현황/             # 패키지사업부
│   └── CLAUDE.md
├── 생산진행현황_패키지/             # 패키지 원본 데이터(읽기 전용)
├── AX 전환 계획/                    # Phase 0 산출물(보고서·발표·대시보드)
├── docs/                            # 인터뷰·메모 정리·도식
├── convert.py / convert_package.py  # 엑셀→JSON 변환기
└── web/                             # 생산현황 조회 서비스 정적 웹앱
```

하위에 묶여 있는 개별 서브 프로젝트:
- **생산현황 조회 서비스** — 엑셀 원본을 JSON으로 변환해 모바일 정적 웹으로 영업사원이 거래처 현황을 조회하는 서비스. (`convert.py`, `convert_package.py`, `web/`)
- **출판사업부 AX 전환** — `출판_생산 진행 현황/CLAUDE.md` 참조
- **패키지사업부 AX 전환** — `패키지_생산진행현황/CLAUDE.md` 참조
- **Phase 0 산출물** — `AX 전환 계획/` 폴더의 보고서·대시보드·발표

- **ERP**: 자체개발 중

## AI 개입 최소화 규칙 (인터뷰·메모 정리 시 최우선)

이 프로젝트(및 모든 하위 폴더)에서 회사 프로세스·문제점·현황을 정리할 때 Claude는 다음을 반드시 지킨다.

1. **출처가 없는 내용을 절대 만들어 적지 않는다.**
   - 사용자가 직접 말한 내용, 사용자가 업로드한 메모/녹음/문서에 있는 내용만 사용한다.
   - "일반적으로 인쇄소는 이렇다", "이런 게 문제점일 것이다" 같은 추론·일반론·상식 기반 추가 금지.
   - CLAUDE.md의 회사 정보(거래처, 설비 목록 등)는 사실 근거로 사용 가능하나, 그것으로부터 문제점을 추론해 적는 것은 금지.

2. **문제점은 사용자가 "문제다"라고 말한 것만 적는다.**
   - 흐름(프로세스 단계)은 메모·인터뷰에 적힌 그대로 옮긴다.
   - 흐름에서 Claude가 스스로 "이게 비효율적이다", "이게 가시성이 낮다"고 판단해 추가하지 않는다.
   - 사용자가 명시하지 않은 단계에는 문제점 칸을 비워둔다. 비어 있는 것이 추측보다 낫다.

3. **불확실하면 반드시 사용자에게 확인한다.**
   - 메모·인터뷰만으로 모호한 부분은 출력에 적기 전에 질문으로 돌려준다.
   - "추정", "아마", "~일 것이다" 표현으로 출력에 채워넣지 않는다.

4. **요약·정리 결과물에는 항상 출처를 함께 적는다.**
   - 각 항목 옆에 출처 표시를 남긴다.
   - 출처 표기 형식 (반드시 이 형식만 사용):
     - 인터뷰에서 나온 내용 → `(YYYY-MM-DD 인터뷰)` 형식. 예: `(2026-04-08 인터뷰)`
     - 손글씨·종이 메모 → `(메모 IMG_xxxx)` 또는 `(메모 YYYY-MM-DD)`
     - 사용자가 채팅에서 직접 말한 내용 → `(사용자 채팅)` 또는 `(YYYY-MM-DD 채팅)`
   - **절대로 "녹음", "녹취", "음성" 같은 표현을 출처에 쓰지 않는다.**
     - 이유: 사내 인터뷰가 녹음되고 있다는 사실이 직원에게 알려지면 솔직한 문제점 발언이 어려워진다.
     - 내부 작업 파일·공유 산출물·PPT·문서 어디에도 "녹음"이라는 단어를 출처로 남기지 않는다.
     - 음성 파일을 전사해서 인용하더라도 출처는 항상 `(YYYY-MM-DD 인터뷰)` 형식으로 표기한다.
   - 출처를 댈 수 없는 항목은 적지 않는다.

5. **이 규칙은 PPT·도식·문서·메모 어떤 산출물에도 동일하게 적용된다.**

## 파일·도구 한계는 즉시 보고한다 (사용자가 모르고 넘어가지 않게)

사용자가 파일을 업로드했거나 작업을 요청했을 때, Claude가 다음 중 하나에 해당하면 **그 사실을 모르는 척 넘어가지 말고 즉시 사용자에게 알린다.**

1. **읽거나 해석할 수 없는 파일을 받았을 때** — 예: 음성 파일을 받았는데 전사 모델이 없거나, 영상/이미지를 받았는데 처리 도구가 없는 경우.
2. **변환·전처리가 필요한 경우** — 예: HEIC → JPEG, m4a → 텍스트, PDF → 표 추출 등이 필요한 경우.
3. **작업 환경(샌드박스)의 제약 때문에 못 하는 경우** — 예: 외부 호스트 차단, 디스크 한도, 모델 다운로드 불가 등.
4. **결과의 일부만 처리 가능한 경우** — "이건 됐고 이건 안 됐다"를 명확히 분리해서 말한다.

이때 반드시 다음을 함께 알린다.
- 무엇을 못 했는지
- 왜 못 했는지(기술적 이유)
- 사용자가 어떻게 하면 가능해지는지(노트북에서 무료 도구로 처리 → 결과만 폴더에 드롭 같은 구체적 대안)
- 사용자가 직접 처리해 준 결과를 받으면 Claude가 곧바로 이어서 무엇을 할 수 있는지

**가만히 추측으로 메우거나, 못 했다는 사실을 숨기고 부분 결과만 내놓는 것을 금지한다.** 사용자가 모른 채로 잘못된 산출물을 받게 되면 이 프로젝트의 신뢰가 깨진다.
