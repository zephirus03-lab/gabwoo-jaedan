## 0. QA 전략 개요

### 0.1 핵심 원칙

이 서비스는 **실제 B2B 고객 응대에 사용**된다. 데이터가 틀리면 거래처 앞에서 신뢰를 잃는다.
따라서 QA의 최우선 목표는 "데이터 정확성"이며, 기능 테스트는 그 다음이다.

```
Priority 1: 데이터 정확성 (엑셀 ↔ data.json ↔ 화면 일치)
Priority 2: 핵심 사용자 흐름 (검색 → 결과 카드)
Priority 3: KPI 카드 정합성
Priority 4: 모바일 레이아웃
Priority 5: 성능 (로딩 속도)
```

### 0.2 서비스 아키텍처 (QA 관점)

```
[3개 엑셀 파일] → [convert.py] → [data.json] → [index.html] → [모바일 브라우저]
     ↑                ↑               ↑              ↑
  (진실 원천)     (변환 위험점)    (중간 산출물)   (사용자 접점)
```

**위험점 순위**:
1. `convert.py` 파싱 오류 → 데이터 손실/왜곡 (최고 위험)
2. `merge_badges()` 품명 매칭 실패 → 배지 오표시 (높음)
3. 공정 텍스트 파싱 불규칙 → 진행 바 오표시 (중간)
4. UI 렌더링 버그 (낮음 — 사용자가 직접 확인 가능)

### 0.3 QA 담당 분리

| 영역 | 담당 | 이유 |
|---|---|---|
| **데이터 정확성 검증** | **사람 (도메인 전문가)** | 원본 엑셀을 읽을 수 있는 사람만 판단 가능 |
| convert.py 기능 테스트 | 개발팀 | 코드 실행 및 결과 확인 |
| UI/UX 기능 테스트 | 개발팀 | 브라우저 조작 |
| 모바일 레이아웃 | 개발팀 | 실기기 테스트 |

## 1. 데이터 정확성 검증 전략

### 1.1 수동 샘플 교차 검증 (최고 우선순위)

**목적**: convert.py가 수주관리 엑셀 데이터를 정확히 읽었는지 확인
**담당**: 사람 (도메인 전문가) — 엑셀과 브라우저를 나란히 열고 비교
**시점**: 첫 convert.py 실행 후, 배포 전 필수

#### 샘플링 방법 (10건)

무작위 선택이 아니라 **경계값 중심 샘플링**:

| 샘플 | 선택 기준 | 확인 목적 |
|---|---|---|
| #1~3 | 가장 최근 수주 3건 | 최신 데이터 정확성 |
| #4~5 | 공정이 가장 많이 진행된 건 (인쇄+코팅+톰슨 완료) | 다중 완료 공정 파싱 |
| #6 | 공정이 하나도 진행 안 된 건 | null 처리 |
| #7 | 생산일 컬럼에 날짜 있는 건 (완료 처리 대상) | completed_at 매핑 |
| #8 | 수량이 가장 큰 건 | 숫자 포맷 보존 |
| #9 | 공정 텍스트가 복잡한 건 (예: "옵셋인쇄,톰슨,단면접착,실크") | 공정 파싱 |
| #10 | 납기가 과거인 건 (지연 건) | 날짜 처리 |

#### 각 샘플에서 확인할 항목

```
□ 관리번호 (mgmt_no)    — 엑셀 D열 vs data.json mgmt_no
□ 거래처 (client)       — 엑셀 E열 vs data.json client
□ 품명 (product)        — 엑셀 F열 vs data.json product
□ 수주량 (qty)          — 엑셀 G열 vs data.json qty (숫자 타입 확인)
□ 납기예정 (deadline)   — 엑셀 I열 vs data.json deadline (YYYY-MM-DD 형식)
□ 공정 7개 날짜         — 엑셀 인쇄~접착 열 vs data.json progress 객체
□ 생산일 → completed_at — 엑셀 생산일 열 vs data.json completed_at
□ 공정 텍스트 파싱      — 엑셀 "공정" 열 텍스트 vs progress 키 목록
```

### 1.2 건수 대조 검증

```
□ 수주관리 엑셀의 진행 중 주문 건수 == data.json summary.in_progress
□ 수주관리 엑셀의 전체 주문 건수 >= data.json orders 배열 길이
□ 인쇄종합 "오늘 인쇄" 건수 == data.json summary.printing_today
□ 출고리스트 최신 시트 건수 >= data.json shipments_today 배열 길이
```

### 1.3 날짜 형식 일관성

```
□ 모든 날짜 필드가 "YYYY-MM-DD" 형식
□ 타임존 오프셋 또는 UTC 변환 오류 없음 (datetime → .date() 필수)
□ updated_at과 generated_at이 오늘 날짜
```

### 1.4 타입 보존

```
□ qty: int (문자열 "10,000" 아님)
□ deadline: str (datetime 객체 아님)
□ progress 날짜: "YYYY-MM-DD" 또는 null (빈 문자열 아님)
□ completed_at: str 또는 null
```

### 1.5 한글 인코딩

```
□ 한글 거래처명/품명 깨짐 없음
□ data.json UTF-8 without BOM
```

### 1.6 Null vs 빈 문자열

```
□ 미완료 공정: null (빈 문자열 아님)
□ 생산일 없음: completed_at = null
□ 배지 미해당: false (null 아님)
```

### 1.7 배지 매칭 검증

```
□ 출고리스트 상위 5건이 shipments_today에 존재
□ 배지 오표시(위양성)보다 생략(false)을 선호
```

## 2. 테스트 시나리오 (20개)

### 그룹 A — 대시보드 기본 (4개)

- **TS-01** 전체 대시보드 로드: 3초 이내 KPI/차트/리스트 표시
- **TS-02** KPI 카드 vs 리스트 건수 일치
- **TS-03** 최근 7일 차트 날짜 필터 (미래 날짜 없음)
- **TS-04** 출고 리스트 최신 날짜만 표시

### 그룹 B — 검색 (6개)

- **TS-05** 관리번호 완전 일치 검색 (예: "260402-0005" → 1건)
- **TS-06** 품명 부분 일치 검색 (예: "클렌저" → 포함 주문)
- **TS-07** 거래처 부분 일치 검색 (예: "코스" → 코스맥스/코스메카)
- **TS-08** 검색 결과 없음 ("ZZZZZ" → "결과 없음" 메시지)
- **TS-09** 검색 초기화 (빈 문자열 → 대시보드 복귀)
- **TS-10** 한글 검색 ("정샘물" → 정샘물뷰티)

### 그룹 C — 주문 카드 엣지 케이스 (4개)

- **TS-11** 공정 0개 완료 주문 (진행 바 모두 빈 상태)
- **TS-12** 모든 공정 완료 주문 (완료 배지)
- **TS-13** 공정 일부 완료 (순서 정확)
- **TS-14** 인쇄 오늘 배지 표시

### 그룹 D — 성능/환경 (3개)

- **TS-15** 대용량 500건 이상 (로드 < 2초, 검색 < 100ms)
- **TS-16** iOS Safari 모바일 뷰
- **TS-17** Android Chrome 모바일 뷰

### 그룹 E — v2 Deferred (3개, v1 블로커 아님)

- **TS-18** 로그인 미승인 → 대기 화면
- **TS-19** 로그인 승인 → 대시보드
- **TS-20** 관리자 신규 사용자 승인
