### Value Delivered (4-Perspective)

| 관점 | 내용 |
|---|---|
| **Problem** | 패키지 사업부 생산 현황이 3개의 엑셀 파일(수주관리/인쇄종합/출고리스트)에 분산되어 있어, 영업사원·생산팀·거래처 문의 대응 시 매번 여러 파일을 열어 확인해야 함. NAS에 있는 파일은 모바일에서 접근 불가. |
| **Solution** | 3개 엑셀을 단일 JSON으로 변환하고, 모바일 우선 대시보드 + 통합 검색으로 한 화면에서 조회. 출판 프로젝트에서 검증된 "정적 사이트 + 수동 변환" 패턴을 재사용하여 빠르게 구축. |
| **Function / UX Effect** | (1) 관리번호/품명/거래처로 단일 검색창 검색 → 공정 진행률 카드 반환 (2) KPI 4개(진행중/오늘인쇄/출고예정/완료) 즉시 확인 (3) 최근 7일 완료 추이 차트 (4) 오늘 출고 예정 요약. 모바일에서 3초 안에 필요한 정보 도달. |
| **Core Value** | 내외부 모두가 동일한 생산 현황을 보는 **단일 진실의 원천(Single Source of Truth)**. v2에서 구글 드라이브 자동 동기화로 진화할 수 있는 기반. |

## 1. User Intent Discovery (Phase 1)

### Q1. 핵심 목적
패키지 사업부의 생산 현황을 모바일에서 조회하는 정적 웹서비스. 출판 프로젝트의 성공 패턴을 패키지 데이터에 적용.

### Q2. 대상 사용자
**D) 사내 통합 대시보드** — 영업사원, 생산팀장, (장기적으로) 외부 거래처 모두 동일한 화면을 공유.

### Q3. 성공 기준
- 영업사원이 모바일에서 거래처 문의를 3초 안에 답변할 수 있다
- 생산팀이 오늘 인쇄 현황을 한 화면에서 파악할 수 있다
- 데이터 정확도 100% (원본 엑셀과 대조 검증 통과)

### Q4. 제약
- **원본 엑셀 파일 절대 수정 금지** (CLAUDE.md 규칙 준수)
- v1은 수동 변환, v2에서 구글 드라이브 자동화
- 실제 고객 응대에 사용되므로 데이터 정합성 최우선

## 2. Alternatives Explored (Phase 2)

| 접근법 | 설명 | 결과 |
|---|---|---|
| **A. 출판 패턴 확장** ⭐ | 출판 프로젝트와 동일 구조, 3개 파일을 하나의 data.json으로 병합 | **채택** |
| B. 통합 데이터 모델 우선 | 관리번호로 모든 데이터 JOIN, 정규화된 단일 주문 모델 | 기각 (출고리스트 매칭 불확실, 복잡도↑) |
| C. 수주관리만 최소 MVP | 1개 파일만 먼저 | 기각 (사용자 통합 요구와 불일치) |

**채택 이유**: 출판 프로젝트에서 이미 검증된 패턴. Vercel 정적 배포, 수동 변환 스크립트 모두 재사용 가능. v2 자동화 시 convert.py만 교체하면 구조 유지.

### 4.4 변환 규칙 요약

**수주관리 (핵심)**
- 월별 시트 패턴 `수주관리 2026. NN월` 자동 탐색, 전체 병합
- 컬럼: No/담당/등록일/관리번호/매출처/품명/수주량/정미량/납기예정/공정/인쇄/코팅/실크,부분uv/박/형압/톰슨/접착/생산일/재작업/비고
- `인쇄~접착` 7개 컬럼에 날짜가 있으면 해당 공정 완료
- `생산일`에 값 → 전체 완료 처리
- `공정` 텍스트 필드에서 실제 해당 주문에 적용되는 공정만 추출해 진행 바에 표시

**인쇄종합 (보조)**
- `인쇄일정관리` 시트만 파싱 (나머지 17개 시트 무시)
- `계획일`이 오늘인 행 → `printings_today`
- `수주번호` 기준으로 수주관리와 JOIN → `orders[].badges.printing_today = true`

**출고리스트 (보조)**
- 시트명 `MMDD` 패턴 중 가장 최근 날짜 시트만 파싱
- `업체명/제품명/출고수량/출고처/비고(상태)` 추출
- 품명 기반 fuzzy 매칭으로 `orders[].badges.shipping_today` 세팅 (정확도 낮으면 배지만 생략)

### 4.5 파일 크기 고려
- 인쇄종합 18.7MB → 필요 시트 1개만 `read_only=True`로 스트리밍 파싱
- data.json 예상 크기: 500KB ~ 1.5MB (연간 누적 1000건 기준)

## 5. Architecture

### 5.1 데이터 흐름
```
[3개 엑셀 (NAS 수동 복사)]
    ↓
[python3 convert.py]
    ├─ parse_orders()      → 수주관리 월별 병합
    ├─ parse_printings()   → 인쇄일정관리 오늘 추출
    ├─ parse_shipments()   → 출고리스트 최신 시트
    └─ merge_badges()      → 수주번호/품명 기준 배지 부착
    ↓
[web/data.json]
    ↓ (git commit → git push)
[Vercel 자동 배포]
    ↓
[모바일 브라우저 fetch('data.json')]
    ↓
[index.html 렌더링]
```

### 5.2 구현 순서
1. `convert.py` 뼈대 (출판 프로젝트의 convert.py 참고)
2. `parse_orders()` — 수주관리 월별 시트 병합
3. `parse_printings()` — 인쇄일정관리 오늘 추출
4. `parse_shipments()` — 출고리스트 최신 시트
5. `merge_badges()` — 배지 부착
6. 원본 엑셀과 샘플 10건 수동 대조 검증
7. `web/index.html` — 대시보드 + 검색 + 결과 카드
8. 로컬 테스트 (모바일 뷰포트)
9. Vercel 배포
10. 실사용 피드백 수집

## 7. Risks & Mitigation

| 리스크 | 영향 | 완화책 |
|---|---|---|
| 출고리스트에 관리번호 없음 → 품명 매칭 실패 | 배지 오표시 | fuzzy match 신뢰도 낮으면 배지 생략, 별도 섹션에 리스트 표시 |
| 인쇄종합 18MB 로딩 느림 | 변환 시간 증가 | `read_only=True`, 필요 시트 1개만 파싱 |
| 수주관리 "공정" 텍스트 파싱 불규칙 | 진행 바 오표시 | 매핑 테이블 유지, 매칭 실패 시 전체 공정 기본 표시 |
| 원본 엑셀 형식 변경 | 변환 실패 | 변환 오류 시 명확한 에러 메시지, 기존 data.json 유지 |
| 데이터 정확도 미검증 | 실 업무 신뢰 상실 | 첫 배포 전 샘플 10건 수동 대조 필수 |
